Teaching

What I value in teaching:

  • Teaching is supposed to challenge students. If students are not being challenged, teach more.
  • I value important concepts and ideas. Clearly let students know what is a main idea and what are details. Help students understand the structure of the subject. Connect new ideas with things the students already know. Create experiences that help make the connetions.
  • I value concrete examples in discussion.

Things that frustrate me about teaching:

  • It is hard to find common ground among CS teachers to discuss important ideas.
  • Curricular choices force us to discuss topics that I do not think are appropriate for beginners. (AP: when does mutating a variable inside a function cause a change visible outside the function?)

Things that frustrate me about discussing teaching:

  • Abstract discussion makes it easy to mis-communicate or talk past each other about important topics.
  • There is little agreement on common goals or what is important.

Things that would make discussing teaching feel more productive to me:

  • Examples of student misunderstandings.
  • Programming exercises that all participants do, then discuss their solutions and potential student misunderstandings.
  • What are the demographics of students that are signing up for our classes? Do we care? Why are there so many boys? Why so many White/Asian? Is it the same at every school? What observations would help us get insight into this? What outreach would be effective? What kind of exit

How to Design Programs made a profound impact on my teaching. The Data-Centric Introduction to Computing is another book in the same tradition. The connection with [the thinking done in algebra][Bootstrap] immediately addressed some of my greatest frustrations teaching programming by explicitly requiring students to do work that shows they understand the problem, creating examples.

In teaching, seeing is believing, so you’ll have to experiment with this yourself.

Professor Krishnamurthi explains this connection in [a video about the Bootstrap algebra teaching resources][3]

Block Programming

Block programming is a surface feature of a programming language, like the choice of semicolons to end lines in C or indentation to indicate nesting in Python.

For an experienced programmer, block programming is slow and it is usually hard to track changes from one version to another if a graphical programming system is used. That said, user interface design is one area where “drag and drop” is still very useful.

For a beginner, working with a block programming interface can be a lifesaver. Having existing parts set up allows a student to concentrate on how to use the tools to complete the job, instead of spending effort at the level of “what tools are available?” Here are some examples of errors that you can make in C or Javascript but not Scratch:

  • mistakenly beliving something is or is not nested;
  • misspelling;
  • mistakes with braces { ... }.

What should a teacher emphasize?

A think an introductory programming class should emphasize a few key concepts.

  • Making choices
  • Variables to remember things
  • Action based on variables
  • Repetition

The comments below are based on using Scrach and Python in introductory classes, but similar principles should apply almost anywhere.

In my experience, a beginner needs specific instruction and practice on “when do you use a variable?” My mantra is: “A variable is the way a computer remembers things.” “Whenever you need to remember something, you should be using a variable.”

Example: Walking or Running

A character moves around the screen, but they can be running or walking. Running the character moves 15 steps in a given direction, while walking the character moves 5.

This leads to creating a variable for the speed.

Untrained beginners do not all immediately come up with the idea of using a speed variable. Exercises like this help them figure out what quantities should be represented by a variable.

Variations: You can add details allowing the player to change the current speed in finer increments, impose a maximum speed which cannot be exceeded, etc.

An aside: even advanced students make what experts consider to be mistakes with variables, usually by having variables to represent dependent quantities. (The correct alternative is to use a function to compute the dependent quantity from the independent ones, so they can never get out of sync.)

Example: Switches and Fuse Box

There are four buttons. When at least two of them are pressed, the light is on; otherwise the light is off. Add in a master fuse to your circuit: when the master fuse is off, the light is off no matter what; otherwise the light is controlled by the other buttons.

This leads to using two variables to control the light. In Scratch, I would suggest that the light have a control like

forever {
  if (master_fuse is on) and (switches_pressed >= 2) {
    set light on
  } else {
    set light off
  }
}
